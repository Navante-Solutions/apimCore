## 1. Decision Title

**Adopt Go (Golang) for the Edge API Gateway (`apimCore`)**

---

## 2. Context and Problem Statement

### 2.1 Problem Description
The Navante ecosystem (`EduCore`, `CampusCore`, `IdentityCore`, etc.) is primarily built entirely on Java 25 and Spring Boot 4. However, the ecosystem required a high-throughput, low-latency API Gateway at the edge to handle tenant routing, API Key validation, Geo-fencing, and burst rate-limiting (up to 2,000 RPS). 

**Problem statement as a question**: Should we build the Edge API Gateway using our baseline technology (Java/Spring Cloud Gateway) or adopt a specialized systems language (Go) designed exclusively for high-concurrency network proxies?

### 2.2 Why This Decision Is Needed
- **Business context**: The gateway must never become a bottleneck for downstream institutional requests.
- **Technical context**: The gateway needs to limit memory overhead and optimize multiplexed socket connections while mitigating DDoS sweeps before hitting the JVM services.

---

## 3. Decision Drivers (Forces)

### 3.1 Technical Drivers
- **Latency & Concurrency (NFR-P-1)**: Need to handle 2,000 requests per second with a P95 latency under 120ms.
- **Resource Constraints**: The gateway must be extremely lightweight (low RAM footprint) to scale horizontally in Kubernetes.
- **Ecosystem Alignment**: Navante developers are Java-focused. Introducing a new language introduces cognitive load.

### 3.2 Alignment to Architecture Principles

| Principle | Alignment | Impact |
|-----------|-----------|--------|
| P-01: Microservices | ✅ Supports | Perfect isolation of responsibilities. |
| P-02: API-First | ✅ Supports | Centralized edge proxy. |
| P-05: Cloud-Agnostic | ✅ Supports | Go compiles to a static binary seamlessly transportable via Scratch Docker images. |

---

## 4. Considered Options

### Option 1: Java (Spring Cloud Gateway)
**Description**: Continue utilizing the primary tech stack (Java 25).

#### Good (Pros)
- ✅ **Developer Familiarity**: 100% of the team writes Java.
- ✅ **Tooling Parity**: Uses the exact same Jacoco, Maven, and Cucumber BDD setups.

#### Bad (Cons)
- ❌ **Memory Footprint**: JVM memory requirements are typically larger than static Go binaries, impacting horizontal scaling densities.
- ❌ **Cold Starts**: Slower startup times compared to C/Go.

---

### Option 2: Go (Golang) Native Gateway (Chosen)
**Description**: Write `apimCore` in Go 1.24+ leveraging zero-dependency YAML configurations.

#### Good (Pros)
- ✅ **Sub-Millisecond Proxying**: Go's Goroutines map exceptionally well to HTTP socket handling.
- ✅ **Micro-Footprint**: The resulting binary is ~20MB and uses minimal idle RAM.
- ✅ **Hot-Reloading**: Trivial implementation of live-reloading configuration (`-hot-reload`) without restarting the service.

#### Bad (Cons)
- ❌ **Stack Fragmentation**: Introduces a second language to the repository. Engineers must learn Go paradigms.

---

## 5. Decision Outcome

### 5.1 Chosen Option
**"Option 2: Go (Golang) Native Gateway"**

### 5.2 Y-Statement 
> **In the context of** terminating aggressive edge traffic for the Navante ecosystem,
> **facing** the need for an ultra-low latency, memory-efficient network proxy,
> **we decided for** Go (Golang),
> **to achieve** static binaries with sub-millisecond P50 response times and zero-downtime hot-reloads,
> **accepting** the trade-off of maintaining a polyglot repository and requiring engineers to learn a second language.

---

## 6. Consequences

### 6.1 Positive Consequences
- ✅ **Performance**: Meets and exceeds the 2,000 RPS burst threshold without cascading timeouts.
- ✅ **Cost Optimization**: Go's low footprint allows the gateway to run on micro-node types, drastically reducing EC2/EKS compute costs.

### 6.2 Negative Consequences (Accepted Trade-offs)
- ❌ **Knowledge Silos**: The backend teams are Java-native. A learning curve exists for maintaining the `apimCore` routing algorithms.
- **Mitigation**: All `apimCore` architecture logic is strictly restricted to routing, metrics, and auth checking. No business logic (e.g., billing, class scheduling) is permitted in Go.
